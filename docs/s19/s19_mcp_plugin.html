s19 MCP Plugin 外部工具接入

MCPClient 发现调用 · 命名空间隔离 · 动态工具池组装

1. 问题：手写工具的瓶颈

痛点: s01-s18 的所有工具都是手写的——bash、read、write、task、worktree。现在你有 Jira API、部署系统、Notion 知识库 3 个外部服务要接入。你不想为每个服务重写一套工具代码。你需要一个标准协议——外部服务只要实现它，Agent 就能直接调用，不管服务用什么语言写的。

6. 新增方法详解

s19 在 s18 基础上增加了 1 个类 + 2 个方法 + 2 个数据结构，聚焦于外部工具接入：

层次方法/类型作用
客户端层MCPClient封装 tools/list 发现和 tools/call 调用。教学版用 Python 函数模拟 server；真实版通过 stdio JSON-RPC 与子进程通信
连接层connect_mcp()查 MOCK_SERVERS → 创建 MCPClient → 注册工具和 handler → 存入 mcp_clients
安全层normalize_mcp_name()所有非 [a-zA-Z0-9_-] 字符替换为 _，防止特殊字符导致命名冲突或注入
组装层assemble_tool_pool()每轮重建工具池：BUILTIN_TOOLS + 所有 mcp_clients 的工具，前缀 mcp__server__tool
数据层mcp_clients全局 dict 追踪所有已连接的 MCP server
数据层MOCK_SERVERS预注册的 mock server 工厂，模拟真实外部服务

三个关键设计细节

设计点说明
无 prompt 缓存s10-s18 用 prompt cache 避免重复序列化。s19 去掉缓存——connect_mcp 后工具池变了，缓存中的工具列表是旧的，继续用会导致模型调用不到新工具。每轮 assemble_tool_pool 重建。
mcp__server__tool 命名所有 MCP 工具加 mcp__{server}__{tool} 前缀，经过 normalize_mcp_name 规范化。不同 server 的同名工具不会冲突。
MCP 工具仅 Lead 可用教学版中 connect_mcp 是 Lead 工具，assemble_tool_pool 也只服务于 Lead 的 agent_loop。Teammate 仍用固定的 8 个子集工具。真实 CC 中子 agent 继承父级 MCP 配置。
