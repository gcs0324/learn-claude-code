s16 Team Protocols 团队协议

结构化握手 · request-response 模式 · 类型校验

1. 两种协议

协议方向用途状态
shutdownLead → 队友体面关机握手：Lead 发请求，队友收尾后确认pending → approved
plan_approval队友 → Lead计划审批：队友提交高风险计划，Lead 审批后动手pending → approved / rejected
核心机制: 两种协议结构完全一样——一方发请求，另一方给回复，请求和回复通过 request_id 关联。有状态机追踪：pending → approved / rejected。

6. 新增方法详解

s16 在 s15 基础上增加了 1 个 dataclass + 4 个方法 + 1 个字典，聚焦于结构化协议握手：

层次方法/类型作用
状态层ProtocolStatedataclass 追踪每个请求的生命周期（request_id / type / status / sender / target / payload）
状态层pending_requestsdict: request_id → ProtocolState，所有进行中的协议请求
路由层handle_inbox_message()按 msg_type 分发：shutdown_request → 回复并停止；plan_approval_response → 注入审批结果
路由层consume_lead_inbox()统一 inbox 消费入口：先路由协议消息更新状态，再返回剩余普通消息
校验层match_response()按 request_id 查找 → 校验 response_type 匹配 → 校验 status==pending → 更新状态
工具层new_request_id()生成唯一 ID（如 req_004281），贯穿全链路的关联键

7. 整体流程变化：s15 → s16

s15 通信（之前）

flowchart LR A["Lead 发消息"] --> B["队友收消息"] B --> C["队友回复"] C --> D["Lead 收回复"]

s16 协议通信（现在）

flowchart TD A["Lead 发请求"] --> B["创建 ProtocolState pending"] B --> C["BUS.send 带 request_id"] C --> D["队友 handle_inbox_message 路由"] D --> E["队友处理 + BUS.send 带 request_id"] E --> F["Lead match_response 校验"] F --> G["更新 ProtocolState approved/rejected"]

三个关键设计细节

设计点说明
request_id 贯穿全链路请求带着 request_id 出去，回复带着 request_id 回来。ProtocolState 按 request_id 索引，保证请求和回复正确关联。
match_response 三重校验① 按 request_id 查找状态；② 校验 response_type 匹配 request_type（shutdown_response 不会意外 approve plan_approval）；③ 校验 status==pending（防止重复处理）。
统一 inbox 消费check_inbox 工具和主循环末尾都调用同一个 consume_lead_inbox()，先路由协议消息更新状态，再返回剩余内容。避免消息被读走但协议状态没更新。
