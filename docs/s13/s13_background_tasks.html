同步 (s12)后台 (s13)

三个关键设计细节

设计点说明
双重判断 模型通过 run_in_background 参数显式声明慢操作（优先）；模型未指定时，关键词启发式兜底。
真实 CC 的 bash 工具 schema 里有 run_in_background: boolean，模型自己决定，不靠关键词猜。
daemon 线程 + Lock threading.Thread(daemon=True) 执行命令，主线程不阻塞。
threading.Lock 保护 background_tasks 和 background_results 字典的并发读写。
通知注入 后台任务开始时注入 placeholder（Agent 知道有任务在跑）。
后台任务完成后，collect_background_results 收集结果，以 task_notification 格式注入下一轮对话。

s12 → s13 的关系: s12 提供任务依赖图（.tasks/ JSON 文件），s13 在此基础上增加后台执行能力。两个系统独立叠加：任务系统管工作的拆解与顺序，后台任务管慢操作的异步执行。Agent 认领一个任务 → 其中的 npm install 丢后台 → 继续认领下一个任务。

9. block 在 agent_loop 中的完整生命周期

每个 block 是 Anthropic API 返回的原生 ContentBlock 对象（block.name / block.input / block.id），从 LLM 返回到最终通知注入，经历 6 个阶段：

flowchart TD A["1. LLM 返回 response.content list of blocks"] --> B["2. 遍历: for block in response.content"] B --> C{"3. block.type == tool_use?"} C -->|"否"| B C -->|"是"| D{"4. should_run_background block.name block.input?"} D -->|"否 快操作"| E["同步: execute_tool block"] E --> F["注入 tool_result 真实 output"] D -->|"是 慢操作"| G["start_background_task block"] G --> H["daemon 线程执行 execute_tool block"] H --> I["注入 placeholder tool_result"] F --> J["5. collect_background_results 收集已完成"] I --> J J --> K["6. 合并注入: user_content = tool_results + task_notifications"] K --> L["messages.append user_content"] L --> M["下一轮 LLM 调用时看到通知"]

通知给模型的方式

路径时机格式模型何时看到
同步结果 工具执行完立即 tool_result（关联 tool_use_id） 下一轮 LLM 调用
后台 placeholder start_background_task 后 tool_result: "[Background task bg_0001 started]" 下一轮 LLM 调用（模型知道有任务在跑）
后台完成通知 collect_background_results 收集到已完成任务 <task_notification> XML 文本块，追加到 user_content 末尾 下一条 user message 中（不关联任何 tool_use_id）
为什么后台结果不用 tool_result？ 因为 tool_result 必须关联一个 tool_use_id，而后台任务的 tool_use_id 在上轮已经用 placeholder 消耗掉了。后台完成通知以 <task_notification> 纯文本形式追加到 user message，模型在下一轮当作上下文读到。
