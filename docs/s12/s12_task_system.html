三个关键设计细节

设计点说明
文件持久化 每个任务一个 .tasks/{id}.json 文件。create / claim / complete 都实时写盘。
跨会话时，Agent 调 list_tasks 扫描目录即可恢复全部进度，无需内存状态。
blockedBy 依赖检查 can_start 是 claim 的前置条件：blockedBy 全部 completed 才允许认领。
缺失的依赖文件视为 blocked（不存在的 ID = 不能开始），避免引用错误 ID 时崩溃。
complete 自动解锁下游 complete_task 完成后扫描所有 pending 任务，找出 can_start 变为 true 的下游任务并报告。
Agent 不需要手动追踪"谁依赖了我"，系统自动通知。

9. s11 vs s12 对比总结

维度s11s12
任务管理无Task dataclass + 8 个方法 + 5 个工具
存储无持久化.tasks/{id}.json 跨会话
依赖无blockedBy 图 + can_start 检查
工具数量bash, read_file, write_file (3)+ create_task, list_tasks, get_task, claim_task, complete_task (8)
状态机无pending → in_progress → completed
认领机制无claim_task + owner 字段
解锁通知无complete_task 自动报告 unblocked 下游
s10 → s12 的关系: s10 提供 prompt 组装（PROMPT_SECTIONS + 缓存），s12 在此基础上扩展工具列表（+5 个任务工具）并新增 .tasks/ 持久化目录。两个系统独立叠加：prompt 组装管 system prompt 的构建时机，任务系统管工作的拆解与追踪。
