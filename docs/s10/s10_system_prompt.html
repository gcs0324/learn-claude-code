6. 缓存机制详解

层级实现作用
s10 教学版 json.dumps(context, sort_keys=True) 做 key，相同 key 直接返回缓存 避免重复拼接字符串（同一轮多次 LLM 调用时 context 不变）
CC 真实版 三层缓存: lodash memoize + section 注册缓存 + API 级 prompt cache boundary 静态 section 命中 global cache，动态 section 每次重建
为什么用 json.dumps 而不是 hash()? Python 内置 hash() 有进程随机化，不适合做稳定 cache key；遇到 list/dict 会报 unhashable type。

8. 新增方法详解

s10 在 s09 记忆系统之上增加了 3 个方法 + 1 个常量，聚焦于将硬编码 SYSTEM prompt 改为运行时组装：

层次方法/常量代码行作用触发时机
定义层 PROMPT_SECTIONS44分段字典：identity / tools / workspace / memory 四个独立 section常量，被 assemble 读取
组装层 assemble_system_prompt()52按 context 真实状态选择 section 并拼接：始终加载 3 个 + 按需加载 memory缓存未命中时被 get_system_prompt 调用
get_system_prompt()73确定性缓存包装：json.dumps(context) 做 key，命中直接返回，未命中调 assembleagent_loop 每轮开始时 + 工具执行后
状态层 update_context()158从真实文件系统和工具注册表推导 context，不依赖消息关键词agent_loop 每轮开始 + 工具执行后

10. s09 vs s10 对比总结

维度s09s10
prompt 方式硬编码 SYSTEM 字符串PROMPT_SECTIONS + assemble_system_prompt
缓存无get_system_prompt（json.dumps 检测 + 缓存）
状态感知无（prompt 固定不变）update_context 从真实状态推导
修改影响改一处可能影响全局每个 section 独立维护
项目切换需重写整个 prompt只改 context 来源，section 结构不变
新增方法11 个（记忆系统）3 个 + 1 常量（prompt 组装）
核心痛点记忆随压缩丢失prompt 硬编码，不可维护
s09 → s10 的关系: s09 提供记忆系统（.memory/ 文件），s10 的 context 会检查 MEMORY.md 是否存在来决定是否加载 memory section。两层协作：s09 管记忆的读写，s10 管 prompt 的组装时机。
